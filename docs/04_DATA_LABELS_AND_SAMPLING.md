# Data, Labels, and Sampling Reference

This document consolidates everything about the PTB-XL data, how the 71 SCP codes become the MI-vs-NORM task, and the sample funnel. It exists so the labeling logic is unambiguous before anyone writes filtering code.

---

## 1. Files you need from PTB-XL

| File | What it gives you |
|---|---|
| `ptbxl_database.csv` | One row per ECG record; the 28-column metadata table (labels, folds, file paths, quality flags). |
| `scp_statements.csv` | **The lookup table** mapping each SCP code → diagnostic superclass + infarction location. You cannot build the MI cohort without this. |
| `records100/` | 100 Hz signal files (referenced by `filename_lr`). Primary signal source. |
| `records500/` | 500 Hz signal files (referenced by `filename_hr`). Only if you switch sampling rate. |

---

## 2. The 28 metadata columns — what to keep

**MUST use (core pipeline):**
- `ecg_id` — unique key linking signals ↔ labels ↔ attributions.
- `scp_codes` — the labels; a stringified dict like `{'IMI': 100.0, 'NORM': 0.0}`. Parse with `ast.literal_eval`.
- `strat_fold` — official split (1–8 train, 9 val, 10 test).
- `filename_lr` — path to the 100 Hz signal.
- `patient_id` — verify no patient leaks across folds.
- (`filename_hr` — only if using 500 Hz.)

**Useful for concordance (Phase 4.3):**
- `infarction_stadium1`, `infarction_stadium2` — MI stage/localization; helps define territory-relevant leads.
- `heart_axis` — minor aid to lead interpretation.

**Useful for QC / exclusions:**
- `validated_by_human`, `second_opinion`, `initial_autogenerated_report` — keep high-confidence, human-validated labels.
- `baseline_drift`, `static_noise`, `burst_noise`, `electrodes_problems` — signal-quality flags; record in data card, consider excluding very noisy records.
- `pacemaker`, `extra_beats` — candidate exclusions (paced rhythm changes morphology, muddies MI concordance).

**Ignore for this project:** `nurse`, `site`, `device`, `recording_date`, `height`, `weight`, `validated_by`, `report`. (`age`/`sex` only if a later subgroup/fairness analysis is added.)

## 3. Mapping the 71 SCP codes to the MI "dog"

The 71 codes are fine-grained diagnostic + form + rhythm statements. **Do NOT train a 71-class model.** Collapse to the 5 diagnostic superclasses via `scp_statements.csv`, then reduce to binary.

**MI superclass = union of these codes** (with approximate whole-dataset label-mentions):
`IMI` (2676), `ASMI` (2357), `ILMI` (478), `AMI` (353), `ALMI` (288), `INJAS` (214), `LMI` (201), `INJAL` (145), `IPLMI` (51), `IPMI` (33), `INJIN` (18), `PMI` (17), `INJLA` (17), `INJIL` (15). → ~6,863 label-mentions.

**NORM** = `NORM` (9514).

Other superclasses (not our positives, used only for exclusion logic): **STTC** (`NDT`, `ISC_`, `STD_`, `ISCAL`, `NST_`, …), **CD** (`LAFB`, `IRBBB`, `1AVB`, `CRBBB`, `CLBBB`, …), **HYP** (`LVH`, `VCLVH`, `RVH`, …). Rhythm/form statements (`SR`, `AFIB`, `PVC`, `STACH`, `PACE`, …) are not diagnostic superclasses.

> **Multi-label caveat:** counts above are *label-mentions*, not records. PTB-XL is multi-label (~21,799 records, ~61,000 total mentions), so one ECG can carry several codes. The true MI-record count comes only from parsing `scp_codes` per record.

---

## 4. Task definition (locked — see Decisions D4, D9)

- **Positive** = record whose `scp_codes` contain any MI-superclass code (above the chosen likelihood threshold).
- **Negative** = record mapping to NORM only.
- **Retain the specific MI subtype code** per positive (needed for territory → lead concordance).
- **Exclude** records that are MI + another superclass from the concordance subset.

**Strict vs. inclusive (compute both, lean strict):**
- **Strict** — MI likelihood = 100 and pure-NORM. Cleaner, more defensible, smaller n. Preferred.
- **Inclusive** — any MI code at any likelihood. Larger n, noisier. Fallback if strict test pool is too small.

---

## 5. Territory → lead mapping (for the concordance metric)

| MI subtype code | Territory | Expected leads |
|---|---|---|
| AMI, ASMI, ALMI | Anterior / anteroseptal | V1–V4 |
| IMI, ILMI, IPMI, IPLMI | Inferior | II, III, aVF |
| LMI | Lateral | I, aVL, V5–V6 |

This mapping IS the ground truth for lead-concordance. It's only possible because we keep the fine-grained code after collapsing to the superclass for training.

## 6. Sample funnel (estimates — replace with real parse)

| Stage | What | Approx. count |
|---|---|---|
| 1 | MI records, superclass, whole dataset | ~5,469 (vs ~9,514 NORM) |
| 2 | Strict clean MI (likelihood=100, non-co-morbid) | ~3,500 |
| 3 | MI in test fold (fold 10 ≈ 10%) | ~355 |
| 4 | **Correctly-classified MI (sens ~0.80–0.88)** | **~300–350** |

**Two evaluation sample sizes (do not confuse):**
- **Automated metrics:** ALL ~300–350 cases × each xAI method ≈ 1,500–1,750 maps. (Large; supports tight CIs.)
- **Cardiologist rating:** small blinded sample ≈ **40–60 panels** (e.g. 15 cases × 4 methods). Independent of the 6-experiment structure and of the full case pool.

**If strict test pool < ~150:** widen to inclusive, or use validation+test folds (9–10). Never touch training folds.

---

## 7. What the Phase 1 script must output

1. MI-vs-NORM record counts under **both** strict and inclusive definitions, **per fold**.
2. Co-occurrence / exclusion counts (MI + other superclass).
3. Signal-quality flag distribution.
4. A written `data/data_card.md` capturing all of the above + preprocessing settings + random seed.

Everything downstream must be traceable to that data card.
